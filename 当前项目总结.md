# 当前项目总结

## 1. 一句话先看懂当前状态

- 当前项目是一个 **Roblox + Rojo 同步的 Luau 项目**。
- **代码必须改本地 VS Code 文件**，再通过 Rojo 同步到 Studio。
- **场景对象、模型、Part、GUI、Prompt 的检查和调整**，优先通过 **Roblox Studio MCP** 做。
- 当前最新交接重点不是继续大改，而是：**先去 Roblox Studio 做实机回归验证**，重点验证家园升级、房屋显隐、放置槽位、收益 UI、解锁 Prompt。

## 5. 当前架构规范

### 5.1 服务端规范

- 统一入口：`src/server/ServerScriptService/Server.server.luau`
- 所有业务模块都在：`src/server/ServerScriptService/Modules/`
- 模块基本都是：
  - `local XxxService = {}`
  - 暴露若干 API
  - 提供 `XxxService.Init()` 初始化入口

### 5.2 客户端规范

- 主逻辑集中在：`src/client/StarterPlayerScripts/Client.client.luau`
- 客户端主要通过两种方式同步状态：
  - `Player Attribute`
  - `ReplicatedStorage.Remotes` 下的 `RemoteEvent`

### 5.3 UI 规范

- UI 很多是手动在 Studio 里做好的，不一定都由代码创建，但在检查时应该确认，不要忽略。
- 节点常用中文名。
- 代码里对现有 UI 采用“优先复用，找不到再 fallback 创建”的方式。

### 5.4 代码规范

- 顶部使用 `--!strict`
- 中文注释较多
- 函数职责写得相对清晰
- 新功能优先拆成独立服务模块，而不是继续把逻辑堆到入口脚本里

## 6. 服务端入口在做什么

文件：`src/server/ServerScriptService/Server.server.luau`

### 主要职责

#### 1）确保 `ReplicatedStorage.Remotes` 存在

使用函数：

- `ensureFolder(parent, name)`：确保 Folder 存在
- `ensureRemoteEvent(folder, name)`：确保 RemoteEvent 存在

创建的远程事件包括：

- `RequestDamageMonster`
- `RequestPromptInteraction`
- `PlayWeaponHitVFX`
- `RequestUseWeaponSkill`
- `RequestPurchaseWeapon`
- `RequestEquipWeapon`
- `RequestPurchaseUpgrade`
- `OpenWeaponShopUI`
- `OpenUpgradeShopUI`
- `PushClientToast`
- `SyncMonsterInventory`
- `RequestTakeOutMonsterItem`
- `RequestClearPlayerData`

#### 2）确保默认武器 Tool 可用

使用函数：

- `ensureWeaponToolHasHandle()`

作用：

- 检查 `StarterPack.Weapon` 是否存在 `Handle`
- 没有就补一个默认 `Part`
- 给武器补属性：
  - `IsWeaponTool = true`
  - `WeaponLevel = 1`

#### 3）初始化所有服务模块

当前初始化顺序：

1. `AppearanceBootstrapService.Init()`
2. `WeaponToolService.Init()`
3. `WeaponAppearanceService.Init()`
4. `DataPersistenceService.Init()`
5. `CharacterService.Init()`
6. `InventoryService.Init()`
7. `HomeService.Init()`
8. `PlacementService.Init()`
9. `MonsterService.Init()`
10. `LavaService.Init()`
11. `MerchantService.Init()`
12. `WeaponService.Init()`

这里有一个很重要的顺序约束：

- **持久化服务必须早于 Home/Placement 初始化**，因为要先把存档读进来。

#### 4）绑定主要远程请求

- `RequestTakeOutMonsterItem` → `InventoryService.TakeOutToTool`
- `RequestClearPlayerData` → `DataPersistenceService.ClearPlayerData`
- `RequestDamageMonster` → `MonsterService.DamageMonster`
- `RequestPromptInteraction` → 依次交给：
  - `PlacementService.TryHandlePromptInteraction`
  - `MonsterService.TryHandlePromptInteraction`

## 7. 功能分区总览

下面按功能区说，方便后续对话快速定位。

### A. 数据与存档系统

相关文件：

- `PlayerData.luau`
- `DataDirtyService.luau`
- `DataPersistenceService.luau`
- `ProfileService.luau`

#### `PlayerData.luau`

作用：

- 管理玩家运行时内存数据。
- 管理金币、家园等级、武器等级、属性升级等级等。

主要函数：

- `Get(player)`：取玩家当前数据表
- `AddGold(player, amount)`：加金币
- `SetGold(player, amount)`：设置金币
- `TrySpendGold(player, amount)`：尝试扣金币
- `GetHomeLevel(player)` / `SetHomeLevel(player, homeLevel)`：读写家园等级
- `AddOwnedWeaponLevel(player, weaponLevel)`：记录已拥有武器等级
- `SetWeaponLevel(player, weaponLevel)`：设置武器最高等级
- `SetEquippedWeaponLevel(player, weaponLevel)`：设置当前装备武器等级
- `AddSpeedLevel` / `AddAttackLevel` / `AddHealthLevel` / `AddCarryLevel`：升级属性
- `Sync(player)`：同步到 attribute / leaderstats
- `Export(player)`：导出给存档模块
- `Cleanup(player)`：玩家离开时清理内存

#### `DataDirtyService.luau`

作用：

- 统一管理“哪些玩家数据脏了，需要存档”。

主要函数：

- `Bind(handler)`：绑定脏数据处理器
- `MarkDirty(playerOrUserId, saveSoonSeconds?)`：标记玩家需要保存

#### `DataPersistenceService.luau`

作用：

- 负责把运行时数据与持久化存档打通。
- 玩家进游戏时读档，离开时保存。

主要函数：

- `Init()`：启动存档系统
- `ClearPlayerData(player)`：清空测试数据
- `GetProfile(player)`：拿底层 profile

关键内部函数：

- `applyLoadedData(player, data)`：把读到的存档应用给各业务系统
- `writeBack(player, profile)`：把内存中的 PlayerData / Inventory / Placement 写回 profile
- `markDirty(userId, saveSoonSeconds?)`：调度保存
- `onPlayerAdded(player)` / `onPlayerRemoving(player)`：进出游戏流程

#### `ProfileService.luau`

作用：

- 这是底层通用存档库，体量大，偏基础设施。
- 一般不把业务逻辑直接写进这里。

你可以把它理解为：

- `DataPersistenceService` 是项目自己的“存档业务层”
- `ProfileService` 是底层“DataStore 封装库”

### C. 放置系统

相关文件：

- `PlacementService.luau`

作用：

- 管理家园槽位的解锁、放置怪、卖出、收回、收益累计、收益显示 UI
- 还负责与 Prompt 交互对接

对接模块：

- `HomeService`
- `InventoryService`
- `PlayerData`
- `DataDirtyService`

主要导出函数：

- `Init()`：初始化所有放置逻辑
- `Export(player)`：导出玩家放置数据给存档
- `ApplyLoaded(player, loadedSlots)`：应用存档数据
- `Cleanup(player)`：清理运行时状态
- `TryHandlePromptInteraction(player, promptInstance)`：处理放置系统相关 Prompt

关键内部函数可按职责理解：

#### 1）站位与交互判断

- `getHRP(player)`：取玩家 HumanoidRootPart
- `isPlayerStandingOnSlot(player, slotPart)`：判断玩家是否站在槽位上
- `isPlayerWithinInteractionDistance(player, targetPart, maxDistance?)`：服务端距离校验

#### 2）收益 UI

- `getIncomeDisplayTemplate()`：找收益显示模板 `BillboardGui`
- `findIncomeDisplayTextLabel(gui)`：找可写文本节点
- `setIncomeDisplayText(...)`：写入怪等级、收益、未领取金币等文案
- `setIncomeDisplayVisibility(gui, visible)`：控制收益显示开关
- `isSlotCurrentlyVisible(slotPart)`：判断槽位是否仍在 Workspace 中
- `ensureIncomeDisplayGui(player, slotPart, collectorPart?)`：给槽位挂收益 GUI
- `refreshIncomeDisplayForSlot(...)`：刷新单槽位收益显示

#### 3）已放置怪的头顶显示

- `getPlacedMonsterHeadTemplate()`：找放置怪头顶模板
- `ensurePlacedMonsterHeadBillboard(rootPart)`：保证头顶 UI 存在
- `setPlacedMonsterHeadBillboardText(rootPart)`：刷新头顶文字

#### 4）怪物外观放置

- `getMonsterAppearanceTemplate(monsterLevel)`：找怪模板
- `stripJointsAndConstraints(root)`：清旧约束
- `prepareAppearanceParts(root)`：把模板调整到可挂接状态
- `weldPartsToAnchor(anchorPart, root)`：焊接模板零件
- `weldAnchorToRoot(rootPart, anchorPart)`：把模板锚点焊到放置根 Part
- `sanitizePlacementRoot(inst)`：找可靠根节点
- `findPlacementTemplateRootPart(inst)`：找模板逻辑根
- `moveHierarchyToTarget(...)`：整体平移/旋转到目标位置
- `getPlatformAlignedPlacementCFrame(slotPart, yOffset)`：根据 Platform 朝向计算放置坐标
- `createPlacedMonsterFromTemplate(slotPart, monsterLevel)`：直接用模板生成放置怪
- `applyMonsterAppearanceToPlacedPart(rootPart, monsterLevel)`：回退外观附加方案

#### 5）槽位与经济逻辑

- `getUnlockCost(slotIndex)`：算解锁价格
- `ensureState(player)`：创建/读取玩家槽位运行时状态
- `canUseSlotByHomeLevel(player, slotIndex)`：家园等级是否允许用这个槽位
- `isPrevUnlocked(player, slotIndex)`：前置槽位是否已解锁
- `addUnclaimedToPlayer(player, placed)`：把未领取收益结算给玩家

#### 6）操作行为

- `trySellPlacedMonster(...)`：卖出放置怪
- `tryStorePlacedMonster(...)`：收回背包
- `tryUnlockSlotFromPrompt(...)`：响应解锁 Prompt
- `createPlacedMonsterPart(...)`：创建真正的已放置怪实体
- `tryPlaceFromEquippedTool(player, slotPart)`：从当前装备 Tool 放怪
- `payoutIfAny(player, slotPart)`：触碰 Collector 时领取收益
- `ensureUnlockPrompt(player, slotPart)`：给槽位创建解锁 Prompt
- `rebuildPlacedMonstersWhenHomeReady(player)`：家园准备好后，按存档重建已放置怪

#### 当前这块最新修复重点

- 解锁提示站位判定改成世界坐标，不再受旋转影响
- 放置区 `Platform` 和收币区 `Collector` 的触碰逻辑已拆分
- 收益 UI 改成 **直接挂 BillboardGui**，不再克隆整块 Part
- 收益模板查找改成全局搜索 `Workspace` 中名为 `收益显示` 的对象

## 12. 当前开发规范，接手时必须记住

### 规则 1：代码改本地，场景动 MCP

- `.luau` / `.lua` / `.json` 改 VS Code 本地文件
- Studio 场景对象、模型、UI、属性，优先用 MCP 检查和调整

### 规则 2：不要直接在 Studio 写脚本

- 这是明确禁止项

### 规则 3：不要用 execute_luau 往 Studio 里直接写代码

- 这也是明确禁止项

### 规则 4：先读现状再修改

排错顺序建议：

1. 先看 `PROJECT_HANDOFF.md`
2. 再看本文件
3. 检查 `default.project.json` 和 `Config.luau`
4. 检查 Studio 场景命名是否符合代码假设
5. 最后再决定改代码还是改场景

### 规则 5：新逻辑优先补注释，保持函数职责单一

- 这个项目已经比较依赖函数分拆与中文注释
- 后续继续维护时，尽量沿用这种风格
